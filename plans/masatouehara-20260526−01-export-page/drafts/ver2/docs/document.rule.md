# ドキュメント作成ルール

`docs/` 配下に置く仕様書の書き方ルール。仕様書自身もこのルールに従う。

## 1. 目的

- 後から個別にリバイスできる粒度に保つ
- 重複を排し、相互参照で繋ぐ
- 200 行以内で「眺めて把握できる」状態を保つ

## 2. ファイル命名

| 種別 | パターン | 例 |
|---|---|---|
| 仕様書 | `<kind>.spec.md` | `dir.spec.md`, `versioning.spec.md` |
| ルール | `<kind>.rule.md` | `document.rule.md` |
| 指示書 | `order.<scope>.md` | `order.export.common.md` |

- `<kind>` は名詞 1 語（kebab-case 可、小文字）
- 1 ファイル 1 系統。複数系統を 1 ファイルにまとめない

## 3. 長さと構成

- **1 ファイル 200 行以内**。超えそうなら系統で分割する
- 章立ては「幹 → 枝 → 葉」の 3 階層まで（`#` → `##` → `###`）
- 4 階層以上が必要なら、その節を別ファイルに切り出す

## 4. 系統の分離

「個別にリバイスできるか」を判断軸に、別ファイルにする。同時に変更されないものは同居させない。

例（本プロジェクトの実例）:

| ファイル | 系統 |
|---|---|
| `dir.spec.md` | ディレクトリ構造・レイヤー責務 |
| `versioning.spec.md` | 識別子・ファイル命名 |
| `cli.spec.md` | CLI 引数・終了コード |
| `runtime.spec.md` | 起動方法・検証手順 |

## 5. ファイル内の章立て

### 5.1 必須セクション

- **冒頭の 1〜2 行**: そのドキュメントが何を扱うかの 1 文サマリ
- **末尾の「非対象 / 今後の検討」**: スコープ外の項目を列挙

### 5.2 番号付け

- 章は `## 1. 〜`、節は `### 1.1 〜` の形式
- 番号を付けることで相互参照しやすくする（`section 3.2 を参照` のように）

### 5.3 表記の使い分け

| 内容 | 表記 |
|---|---|
| 項目が並ぶ仕様 | 表（パイプテーブル） |
| パス・コマンド・コード片 | コードブロック ` ``` ` |
| ファイル/パス名の本文中言及 | バッククォート `` ` `` |
| ディレクトリ構造図 | コードブロック内のツリー |

## 6. 相互参照

- 同 `docs/` 内の他ファイルは相対リンクで繋ぐ: `[versioning.spec.md](./versioning.spec.md)`
- 特定の節を指す時はアンカーを付ける: `[versioning.spec.md](./versioning.spec.md#2-version_id)`
- 同じ内容を複数ファイルに書かない。**1 箇所に書いて他から参照**する

## 7. 書き味

- 結論先出し。前置きは 1 行
- 「なぜそう決めたか」が非自明な時だけ理由を 1 行添える
- 例示は最小ケースで 1 つ。網羅しない
- 文末は「である調」「だ調」「ですます調」のどれかに統一（混在しない）

## 8. 非対象 / 今後の検討

- 自動 lint（行数制限・章立て制約の機械チェック）
- テンプレファイルの用意（`spec.template.md` 等）
- 多言語対応（日本語のみで運用）
